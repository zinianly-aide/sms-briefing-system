# Architecture Decisions

## ADR-001：采用前后端分离单仓库结构
- **状态**：Accepted
- **决策**：在同一仓库下维护 `frontend/` 与 `backend/`，文档集中放在 `docs/`。
- **原因**：当前项目规模较小，单仓库更适合快速联调、统一提交与阶段性演进。
- **影响**：后续如需拆分部署，可保留接口契约与文档，按模块再拆仓或拆流水线。

## ADR-002：本地开发默认使用 MySQL 8 容器
- **状态**：Accepted
- **决策**：提供 `docker-compose.yml` 启动本地 MySQL，建表脚本直接复用 backend 中的 `schema.sql`。
- **原因**：降低新成员启动门槛，避免每人手工安装/初始化数据库。
- **影响**：开发环境依赖 Docker；生产环境数据库部署方式可以独立于本地方案。

## ADR-003：后端配置优先走环境变量覆盖
- **状态**：Accepted
- **决策**：`application.yml` 提供默认值，并允许通过环境变量覆盖数据库连接与端口。
- **原因**：避免把机器相关配置写死，便于本地、测试、未来容器化部署复用。
- **影响**：README 与 backend 文档需同步说明变量约定。

## ADR-004：当前阶段保留 Mock + 渐进式持久化
- **状态**：Accepted
- **决策**：在阶段 3 之前允许部分模块继续使用 Mock 数据，但数据库表结构先行落地。
- **原因**：先保证界面与流程可演示，再逐步切换到真实持久化实现，节奏更稳。
- **影响**：阶段 3 的首要工作是替换 MockDataService 对主数据模块的依赖。
